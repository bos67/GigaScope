# Демо-сценарий MVP

## Цель

Показать заказчику, как GigaScope + Ouroboros обнаруживают, анализируют
и объясняют инциденты в банковской инфраструктуре в реальном времени.

## Сценарий: «Падение PaymentService»

### Подготовка (перед демо)
1. Neo4j заполнен: 8 сервисов, ~15 связей DEPENDS_ON/SUBSCRIBES_TO
2. 5 исторических инцидентов с разными паттернами
3. 3 гипотезы в Hypothesis Registry (разные confidence)
4. Event Producer настроен на эмуляцию сбоя

### Демо (5 минут)

**Шаг 1. Показываем здоровую систему**
- GigaScope 3D-граф: все узлы зелёные
- Данные идут из Neo4j через GigaScope API
- Легенда: NAO (сеть), SM (сервисы), МЕТА (данные)

**Шаг 2. Событие: PaymentService начинает тормозить**
- Event Producer запускает сценарий: latency spike
- GigaScope API принимает событие, пишет в Neo4j как Symptom
- Узел PaymentService меняет цвет на жёлтый (warning)
- В боковой панели: «Response time +340%»

**Шаг 3. Каскад: падает PaymentService → TransactionService**
- Processing строит PROPAGATES_TO связи
- TransactionService (зависит от PaymentService через Kafka) начинает фейлить
- Оба узла красные
- Ouroboros travers'ует граф: «PaymentService упал первым. TransactionService — следствие»

**Шаг 4. Ouroboros создаёт гипотезу**
- Traversal находит: рост времени ответа → timeout → падение
- Создаётся гипотеза: «Database connection pool exhaustion вызвал каскадный сбой»
- Confidence: 0.65 (на основе 3 похожих паттернов из истории)
- Гипотеза появляется в Hypothesis Lab на дашборде

**Шаг 5. Человек подтверждает/отклоняет**
- Ты (или оператор) смотришь гипотезу
- Подтверждаешь: «Да, была нагрузка на Database до падения»
- Confidence гипотезы растёт до 0.82
- Гипотеза переходит в статус confirmed

### Сценарий 2: «Паттерн по пятницам» (дополнительно)
- Ouroboros находит, что 4 из 5 последних пятниц в 18:00–20:00
  у CDN-подобного узла растёт latency
- Создаёт гипотезу с confidence 0.68
- Рекомендация: «Проверить конфигурацию перед пиковой нагрузкой в пятницу»
